# Data model

This document defines logical tables and invariants. Migration SQL is the executable
form and must remain consistent with it.

## Identity and access projection

OAuth provider storage is kept separate from the domain schema. The domain stores
only durable identity projections needed for ownership and audit.

### `principals`

`id`, `provider`, `provider_subject`, optional email, display name, and created time.
The provider subject is authoritative; production passkey owners do not require an
email address.

### `workspaces` and `memberships`

A workspace has an opaque ID, name, and creation time. Membership maps a principal
to `owner`, `writer`, or `reader`. Wikimemory creates one workspace and one owner but tests
isolation and roles with additional fixtures.

Uniqueness is enforced for `(provider, provider_subject)`, normalized verified email
per provider where present, and `(workspace_id, principal_id)` membership.

## Documents and revisions

### `documents`

- opaque `id` generated by the service;
- `workspace_id`;
- immutable `type`, `slug`, and `created_at`;
- unique `(workspace_id, slug)`.

Identity fields cannot be updated. Slug rename/alias support is deferred.

### `revisions`

- opaque `id`, `workspace_id`, and `doc_id`;
- server integer `revision_number`, unique per document;
- nullable `parent_revision_id` (null only for revision 1);
- full `title`, `body`, and optional `summary` snapshot;
- `created_at`, authenticated `principal_id`, OAuth `client_id`, optional
  caller-supplied `agent_label`, and required `reason`;
- `operation_id` and canonical `request_hash`;
- nullable `restored_from_revision_id`; and
- unique `(doc_id, revision_number)` and `(workspace_id, operation_id)` through the
  operation record described below.

The database validates that the parent equals the highest existing revision and that
the supplied number is exactly one greater, transactionally. Composite references
ensure the revision and document belong to the same workspace.

### `revision_metadata`

Full-snapshot metadata rows: revision ID, key, value, and declared cardinality.
Singleton keys have at most one row per revision. Multivalued values are unique per
key/revision.

### `revision_links`

Full-snapshot explicit and derived references: revision ID, kind, target slug,
nullable resolved target document ID, and origin (`explicit` or `body`). Explicit
links require a target at write time. Unresolved body references remain immutable;
read queries may dynamically report them as resolved if a matching document now
exists.

Resolved link targets use composite workspace/document references. A constraint plus
write-time validation ensures a resolved target ID and target slug name the same
document in the same workspace.

## Operations and audit

### `operations`

Records workspace ID, operation ID, request hash, authenticated principal, kind,
status, result reference, and creation time. `(workspace_id, operation_id)` is unique.
A repeated operation succeeds only when the stored principal, kind, and request hash
match exactly.

Failed validation before a transaction is not recorded. A committed mutation and
its completed operation record are atomic.

Purge converts affected operations to content-free tombstones retaining operation ID,
principal, kind, request hash, and terminal `purged` status. A matching replay returns
`gone`; it never runs again.

### `audit_events`

Append-only events contain an opaque ID, workspace ID, event kind, server timestamp,
principal ID, OAuth client ID, agent label, document/revision IDs when safe, request
ID, and sanitized JSON detail. Search-query events contain token/result counts, not
the query text.

Audit events have no cascading foreign keys so a purge can preserve its sanitized
record.

### `purge_authorizations`

One-use rows created only after web reauthentication and exact confirmation. A row
contains workspace/document/request IDs, principal, expiry, and a request hash.
Delete guards require a valid row. It is consumed in the purge transaction.

## Search

`current_fts` is an FTS5 virtual table containing workspace ID and document ID as
unindexed fields plus current title, summary, and body. A successful revision
transaction replaces that document's FTS row. Purge removes it.

Recall filters structured fields through indexed relational tables and uses FTS5 for
ranking/snippets. User terms are quoted and combined without exposing FTS operator
syntax.

## Current-state reads

Current reads select the greatest revision number per document and only then load
metadata and links for that revision. No timestamp determines current state.

An archived document is ordinary current state with singleton metadata
`status=archived`. Archiving appends a full revision and preserves searchability and
history; lint omits archived documents. Removing or changing that status through a
later expected-revision update reverses the archive.

## Restore

Restore reads the chosen historical snapshot, applies it as a new revision whose
parent is current, records `restored_from_revision_id`, and uses the same mutation
and conflict machinery as ingest.

## Purge

Purge explicitly deletes one document's FTS row, snapshot children, revisions, and
document in that order, while its authorization remains valid. It replaces associated
operation results with content-free tombstones and never deletes audit events or
identity/workspace rows. The purge authorization and sanitized audit event are part
of the transaction; the authorization is consumed last.

## Export ordering

Lossless JSONL begins with a manifest and emits sanitized principal projections,
workspaces, memberships, document identities, revisions in document/revision-number
order, metadata, links, and sanitized audit events. Exported actor and client IDs are
stable archive-local aliases rather than authentication identifiers. Provider
authentication records, browser sessions, token material, ordinary completed
operations used solely for replay, and purge authorizations are excluded.
Content-free purge operation tombstones are exported so a future reviewed restore
tool can preserve the non-resurrection invariant.
